# Happy Match 工程恢复与胜利宝箱升级设计

**日期：** 2026-06-28

## 目标

把 `fix-empty-play-scene` worktree 中尚未提交的空场景恢复修复迁回当前 `HappyMatchGame`，并在不改变 9x8 三消、33 个目标、特殊道具和胜负规则的前提下，把胜利演出升级到面试展示质量。

## 约束与证据

- 参考 worktree 全程只读；操作前后对排除 Unity 缓存目录的文件做 SHA-256 汇总校验。
- Git 历史确认当前 `main` 已包含新项目的 8 个提交，真正缺失的是参考 worktree 的 1 个 Bootstrap 修改、1 个 Editor 场景守卫和 1 个 EditMode 测试。
- 计划书是最低功能合同；示例视频提供 820x1022 等比竖屏构图、冬夜雨雪、目标栏、棋盘和胜利节奏参考。
- 玩法数据与状态机不改；只增加启动保护、表现资源、演出编排、自动验收证据。

## 方案选择

考虑了三种宝箱方案：继续旋转当前平面盖子、引入完整 3D 宝箱模型、使用一致的闭合/半开/全开 2D 状态序列。选择第三种：它能在现有 2D SpriteRenderer 架构中明确表现“后沿水平铰链、前沿上抬、盖子始终附着”的正确开法，同时避免新增 3D 管线和镜头兼容风险。

## 恢复设计

`Bootstrap` 在运行时加载到无路径的临时空场景且 Build Settings 中存在可玩场景时，先加载 `SampleScene`，否则按原逻辑补齐 Camera 与 GameManager。Editor 侧 `HappyMatchSceneGuard` 只在非批处理、非播放、非编译状态下识别“空场景或仅 Main Camera 的未保存场景”，自动打开磁盘上的可玩场景。两层保护互不依赖。

## 胜利演出设计

宝箱使用五张同风格透明 Sprite：闭合、约 20 度微启、约 50 度半开、约 78 度大开、约 105 度全开。每个状态通过 alpha 边界校准保持箱体底边和宽度一致，状态切换以短交叉溶解和轻微缩放完成：锁扣蓄力 -> 微启漏光 -> 半开 -> 大开 -> 全开回弹。这样玩家读到的是绕后沿铰链的连续刚体翻盖，不再看到侧边竖板或明显跳帧。

全开瞬间先出暖金色口内光和椭圆冲击环，再触发星芒、金币/宝石弧线与小幅镜头脉冲。`Great!` 使用两层描边和弹性入场；背景只适度提亮，保证宝箱仍是视觉焦点。操作按钮在主体演出结束后淡入，避免抢戏。

## 资源与回退

五张 Sprite 使用内置图像生成工作流制作，绿幕移除后以 alpha PNG 存入 `Assets/Resources/VictoryChest/`。资源导入为 Sprite、关闭 mipmap、启用 alpha transparency。若资源无法加载，运行时保留现有程序化宝箱作为回退，避免构建或演示中断。

## 测试与验收

- EditMode：空场景判断、宝箱阶段选择、阶段透明度总和、阶段端点与时序。
- Unity 批处理：全量 EditMode 测试、Mac Standalone 构建。
- 自动演示：重新生成初始、特殊道具、目标飞行、胜利截图和完整视频；胜利截图必须显示全开宝箱而非竖直侧板。
- 源 worktree：前后 SHA-256 汇总完全一致。
